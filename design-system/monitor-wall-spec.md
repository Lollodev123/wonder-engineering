# Monitor wall — geometric & behavioral contract

This is the **target design contract** for concepts generated by the command workflow. It guides mapping and review; it is not a physical engineering specification.

> Source reference media is excluded from the public snapshot. [`reference/visual-anchors.md`](./reference/visual-anchors.md) preserves anonymized visual observations.

---

## Scene (the room around the wall)

The wall lives in a **dark gallery space**, never on a white background. Every reference video is shot in the same ambient: matte dark wall behind, ground reflection beneath, dramatic lighting from above + sides. Often a person silhouette stands to one side for scale.

- **Wall background**: matte dark, `#1A1A1F`. Mild vignette toward edges.
- **Ground**: subtle reflection of the wall, ~18% intensity, fades after one tile-width.
- **Lighting** (3-point):
  - Key light from above-front (`#FFFFFF`, intensity 0.45) — picks out screen edges.
  - Fill from upper-left (`#5C68B5` cool blue, intensity 0.20) — the gallery feel.
  - Rim from upper-right (`#D86937` warm orange, intensity 0.10) — separates monitors from wall.
- **Optional person silhouette**: ~1.75 m tall, navy on dark, standing to the right at ~70% canvas width. Adds *live event* signal. Toggle via brief.

## Grid

- **5 columns × 3 rows = 15 monitors.**
- Monitor aspect: **16:9** (`1.778`).
- Inter-monitor gap: **24 px**.
- Frames: ultra-thin (≤ 1 px) outline OR no outline — the screen content's gradient *is* the boundary. Reference videos use no visible bezel; the screens float as luminous panels.
- Centered horizontally + vertically in canvas.

## Coordinates `(col, row)` — zero-indexed

```
(0,0) (1,0) (2,0) (3,0) (4,0)     row 0  ← top
(0,1) (1,1) (2,1) (3,1) (4,1)     row 1  ← middle (anchors)
(0,2) (1,2) (2,2) (3,2) (4,2)     row 2  ← bottom
```

## Anchors (never detach)

```
✓ ✓ ✓ ✓ ✓                       row 0   (all detach-eligible)
✓ ✗ ✗ ✗ ✓                       row 1   (cols 1-3 are anchors)
✓ ✓ ✓ ✓ ✓                       row 2   (all detach-eligible)
```

3 anchors: `(1,1) (2,1) (3,1)` — the horizontal spine in row 1. They hold the wall together and carry idle content (typically `IMAGE-SLICE` of a tile-spanning hero image, or a heavyweight `NUMBER`).

**12 outer-ring tiles are detach-eligible.** Stage ② peels exactly **6** of them; the other 6 stay in wave with their idle content.

## The three hard motion rules (targeted + reviewed)

1. **No collision.** Generated layouts should avoid visible overlap. The mapper applies schema rules and the critic reviews the rendered loop. The public snapshot does not contain a continuous-path geometry validator.
2. **Vertical-or-sine motion only.** Idle motion is `y_offset = amplitude * sin(...)`. No idle drift in x. Lateral motion appears *only* during a deliberate Stage ② transition or reverse Return — never as ambient.
3. **Phase-shifted but synchronized.** All tiles share `period = 4.0 s` and `amplitude = 18 px`. Per-tile phase is deterministic: `phase = col * 0.45 + row * 0.18` rad.

See [`motion-principles.md`](./motion-principles.md) for the rationale and current validation boundary.

## Wave physics (idle + receded)

Each monitor's center y, in pixels, at time `t`:

```
center.y(col, row, t) = topMargin
                      + row * (monitorH + gap)
                      + monitorH/2
                      + amplitudePx * sin( 2π * t / periodSec
                                           + col * phaseStepPerCol
                                           + row * phaseStepPerRow )
```

- During Stage ① (idle): `amplitudePx = 18`, `periodSec = 4.0`, `phaseStepPerCol = 0.45`, `phaseStepPerRow = 0.18`.
- During Stage ③ (receded background): multiply `amplitudePx` by `0.4` and `periodSec` by `1.6` — the wave continues but settles into the background.

`center.x(col)` = `leftMargin + col * (monitorW + gap) + monitorW/2`. **Constant.** No x drift in idle.

### Idle-wave clearance estimate

Adjacent columns remain horizontally separated in the idle grid. For adjacent rows in the same column, a conservative relative-displacement bound is `2 × amplitude`. With the documented row pitch, the default `18 px` amplitude leaves substantial visual clearance. The Python motion lab still samples the actual primitive functions because grouped and staggered gestures do not all follow the same phase equation.

Treat amplitude changes as a new validation case: run the sampled diagnostic and inspect the rendered loop. Neither step certifies physical clearance.

## Allowed monitor content types

Each monitor displays exactly one of:

| Type | What it is | Visual treatment |
|---|---|---|
| `GRADIENT` | Vibrant gradient fill (idle ambient default) | One of `screen.gradients` from tokens.json. Gradient angle changes slowly over the loop (linear t-driven) — that's part of the "alive" feel. |
| `IMAGE-SLICE` | One slice of a tile-spanning image | A larger product image (e.g. eyewear, molecule, ski jump) is mapped across multiple adjacent tiles. Each tile shows its slice. The wall reads as one image, fragmented. |
| `NUMBER` | Big single numeric (e.g. `5×`, `240+`) | White or accent color, fills 50–60% of tile height. Subtle drop shadow. |
| `TEXT` | Headline + sub | White text, left-aligned at 16 px padding. Optional pill background. |
| `ICON` | Single brand glyph | Stroke-only, white or accent. |
| `CALLOUT` | Floating role-tag pill — Stage ③ only | Black-translucent pill with white text, ~120×40 px. Often paired with a connection line to the focused tile. |

Mix forbidden — one tile, one type. *Tile-spanning images* are the exception: multiple adjacent tiles share `IMAGE-SLICE` of one source image.

## Stage ② reveal kinematics (1.5 s window)

For each of the 6 detaching tiles:

1. **Lift** (0 → 30% = 0 – 0.45 s):
   - `scale: 1.0 → 1.05`
   - `opacity: 1.0 → 0.7`
   - `position: stays at grid slot` (still wave-active)
2. **Travel** (30 → 80% = 0.45 – 1.20 s):
   - `position: lerp(gridSlot, landingSlot)` with `cinematic` easing
   - `scale: 1.05 → 1.10`
   - `opacity: 0.7`
   - **Path review**: the generated artifact should check its own planned path where possible, then the rendered transition must be inspected.
3. **Land** (80 → 100% = 1.20 – 1.50 s):
   - `scale: 1.10` (settles)
   - `opacity: → 1.0`
   - **Content swap**: idle GRADIENT/IMAGE-SLICE may swap to feature content (NUMBER, TEXT, ICON, CALLOUT). The swap is instantaneous on land — never mid-flight.

All 6 detaching tiles share the same timeline. ±80 ms phase variance OK; beyond that = broken sync.

## Stage ③ zoom kinematics (1.6 s window)

- **Selected (focused) tile** (rotates per loopIndex): `scale: 1.10 → 4.00`, `position: → canvas center`, `opacity: 1.0`. Easing: `cinematic`.
- **Other 5 detached**: `scale: 1.10 → 0.85`, `opacity: 1.0 → 0.30`. Lateral position unchanged.
- **9 anchors (still in wave)**: amplitude `× 0.4`, period `× 1.6`, opacity `→ 0.5`.
- **Floating callouts** (optional): role-tag pills + dashed connection lines fade in at `t = 5.2` (last 0.4 s of REVEAL). Connect callouts to features on the focused tile; keep the labels generic in public examples.
- **Quote overlay** (optional): if brief.stage_3_quote is set, fade in at `t = 5.4` (last 0.2 s).

## Return kinematics (0.4 s)

Reverse the outbound paths. The selected tile shrinks from 4× to 1.0 and slides back to its slot. Receded tiles return to scale 1.0 and opacity 1.0. Anchor motion returns to default amplitude and period. Capture the loop and inspect the seam; matching endpoints in code does not guarantee a visually invisible transition.

## Tap trigger

- **Auto-fire** at `t = 2.5 s` per loop (so `loop.webm` always shows the full sequence).
- **Manual button** at canvas bottom-center: `Explore →` pill (white text on translucent black). Click resets `loopT = 2.5` and replays from Stage ②.

## Performance budget

- **60 fps target.** If frame rate dips below 50 for at least one second, throttle in this order: gradient flow → callout fade → wave amplitude.
- **Canvas**: 1920×1080 logical, responsive to viewport. `pixelDensity 1` for capture, `2` for live preview.
